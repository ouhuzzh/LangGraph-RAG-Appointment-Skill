# AI 伴诊平台开发计划

## 一、项目目标

基于当前 Agentic RAG 项目，二次开发一个面向患者的 AI 伴诊后端服务，MVP 阶段优先实现以下能力：

- 医学常识问答
- 智能推荐科室
- 模拟预约挂号
- 模拟取消挂号
- 多轮对话记忆
- 父块/子块分层检索
- 多 Agent 协同处理复合任务

## 二、当前约束与范围

### 1. 已确认需求

- 挂号与取消挂号先做 mock，后续再接数据库和真实接口
- 文档输入为多种格式
- 分导诊阶段当前只输出推荐科室
- 多 Agent 同时用于复合任务并行和复杂医学问答
- 项目优先按后端服务方式设计
- 第一优先级是医学问答主链路

### 2. MVP 范围

#### 本阶段要做

- 后端服务架构
- Redis 短期记忆
- PostgreSQL + pgvector 数据存储
- 多格式文档解析与入库
- 父块/子块分层索引
- Query Rewrite
- 重排序 Rerank
- 上下文压缩与 Fallback
- 意图路由
- 推荐科室
- mock 挂号 / 取消挂号
- 多 Agent 聚合处理

#### 本阶段暂不做

- 真实医院 HIS 接口
- 医生端工作台
- 正式诊断结论
- 处方建议
- 管理后台

## 三、整体架构目标

目标架构如下：

```text
客户端 / 前端
    ->
API 接入层
    ->
会话服务
    -> Redis 短期记忆
    -> PostgreSQL 长期摘要
    -> LangGraph 调度器
         -> 医学问答流
         -> 推荐科室流
         -> 预约挂号流
         -> 取消挂号流
         -> 复合任务多 Agent 流
    ->
知识库检索层
    -> PostgreSQL 父块
    -> pgvector 子块检索
    ->
业务工具层
    -> mock 挂号
    -> mock 取消挂号
```

## 九、验收标准

### 医学问答

- 能基于知识库回答医学常识问题
- 回答来源于检索到的文档上下文
- 支持 Query Rewrite、父块回填、Fallback

### 记忆

- Redis 能恢复最近多轮对话
- 长对话不会因为上下文过长而失控

### 推荐科室

- 能根据症状描述输出推荐科室
- 信息不足时会先追问澄清

### 挂号/取消

- 能调用 mock 工具完成预约和取消
- 返回结果结构化、可解释

### 多 Agent

- 复合任务可以拆分处理并聚合回复

## 十、主要风险与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 多格式文档解析质量不稳定 | 检索效果差 | 做解析器分流，并保留兜底解析方案 |
| pgvector 检索效果不理想 | 医学问答质量下降 | 引入 Rerank 和全文检索补召回 |
| 路由不稳定 | 走错流程 | 优先使用规则层拦截明显场景 |
| 多 Agent 延迟较高 | 用户体验下降 | 仅在复合任务和复杂问题时启用并行 |
| 多轮记忆成本过高 | Token 成本上升 | 使用摘要压缩 + 短窗口记忆策略 |
